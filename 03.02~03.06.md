경민님 안녕하세요! 검토 결과 공유 드립니다.  
  
**1. 콘텐츠 제공 범위**  
현재 저희 쪽에서 Content API/MSK로 제공 가능한 대상은 cms2 (/content/{content_id}) 한정입니다.  
문서에 언급된 plan, live, campaign, brand 등 다른 source_type은 콘텐츠가 아닌 소재모델링에서 제공 가능한 범위 입니다.  
  
 **2. 카테고리 ID 관련**  
요청 스키마의 category_id는 상품 카테고리가 아닌, 콘텐츠 도메인 자체의 카테고리 체계(에디션/드롭/매거진 등 콘텐츠 유형 분류)에 해당합니다. 콘텐츠 API에서 상품 카테고리 정보는 별도로 관리하고 있지 않습니다.  
  
**3. MSK 이벤트 기반 동기화**  
이벤트 방식의 데이터 수신은 신규 개발이 필요한 영역이라, 팀 내 우선순위 조율이 선행되어야 합니다.  
희망하시는 1분기 일정 내 될지는 확인이 필요합니다  
  
**4. 배열 데이터 형식 (brand_id, product_ids 등)**  
콤마 구분 String 형태(“1123,123,4343”)로의 제공은 어렵습니다. 이벤트 개발 시 P13N팀 전용 스펙이 아닌 범용적인 형태로 설계되어야할거 같습니다. 다른 도메인에서도 이벤트 방식 전환을 고려중인 영역이 있어 동일한 이벤트를 함께 소비하게 될 가능성이 높습니다. 그래서 스펙은 표준화된 구조로 가져가야 할 것 같습니다.
