# simulation-engine — 대화형 인터뷰 시뮬레이션 규약

## 개요

`user-simulator` 에이전트가 가상 인터뷰를 진행할 때 참조하는
질문 뱅크, 인터뷰 흐름 제어 로직, 품질 검증 규칙을 정의하는 보조 스킬.

---

## 1. 인터뷰 질문 뱅크

### 1-A. Warm-up 질문 (단계당 2개 선택)

페르소나의 배경과 현재 상태를 파악하여 신뢰 관계(Rapport)를 형성한다.

```
[WU-01] 요즘 {도메인} 관련해서 어떻게 지내세요? 많이 쓰시는 편인가요?
[WU-02] {도메인}을 처음 접하게 된 계기가 있으셨나요?
[WU-03] 보통 {도메인 관련 행동}을 얼마나 자주 하시는 편이에요?
[WU-04] 주로 어디서/어떤 상황에서 {도메인} 관련 것들을 찾아보시나요?
[WU-05] {도메인} 쪽에서 요즘 뭔가 불편하거나 아쉬운 게 있으세요?
```

### 1-B. Problem Probe 질문 (단계당 3개 선택)

현재 사용자의 실제 Pain을 구체적 상황으로 이끌어낸다.
**추상적 의견이 아닌 실제 경험을 묻는 것이 핵심.**

```
[PP-01] 마지막으로 {문제 상황}을 겪었을 때 어떤 상황이었는지 자세히 들려주실 수 있어요?
[PP-02] 그 상황에서 어떻게 해결하셨어요? 만족스러우셨나요?
[PP-03] 지금 {문제}를 해결하기 위해서 주로 뭘 사용하세요? (앱, 검색, 지인 등)
[PP-04] 그 방법에서 가장 번거롭거나 아쉬운 부분이 있다면요?
[PP-05] 그 문제 때문에 포기하거나 다른 걸 선택한 적도 있으세요?
[PP-06] 비슷한 상황이 얼마나 자주 생기는 편이에요? 그럴 때마다 어떻게 하세요?
[PP-07] 주변에서 같은 문제로 불편해하는 분들 보신 적 있으세요?
```

### 1-C. Hypothesis Test 질문 (PM 가설 기반)

PM의 핵심 가설을 검증한다. 직접적 의견 질문보다 가상 상황을 제시한다.

```
[HT-01] 만약 {가설 기능/솔루션}이 있다면 어떻게 쓰실 것 같으세요?
[HT-02] 그런 서비스를 봤다면 한 번 써볼 것 같으세요? 주저하게 되는 부분이 있다면요?
[HT-03] {가설 기능}이 {가격}이라면 어떠세요? 합리적이라고 느끼시나요?
[HT-04] 지금도 비슷한 걸 다른 방식으로 하고 계신 게 있나요?
[HT-05] {가설 기능}이 실제로 생긴다면 지금 방식 대신 쓸 것 같으세요, 아니면 병행?
[HT-06] 이런 기능이 없어서 그냥 포기했거나 다른 걸 썼던 경험이 있으세요?
```

### 1-D. Solution Probe 질문 (이상적 해결책 탐색)

```
[SP-01] 지금 방식에서 딱 하나만 바꿀 수 있다면 무엇을 바꾸시겠어요?
[SP-02] 이 문제를 완벽하게 해결해주는 무언가가 있다면 어떤 모습일 것 같으세요?
[SP-03] 이 분야에서 정말 잘 만들었다고 생각하는 서비스나 경험이 있으세요?
[SP-04] 친한 친구가 이 문제로 고민하면 뭘 추천해 주시겠어요?
```

## 2. 인터뷰 흐름 제어 규칙

### 2-A. 분기 조건

| 상황 | 처리 방법 |
|------|----------|
| 페르소나가 문제를 경험하지 않음 ("저는 별로 불편하지 않아요") | Warm-up 질문 변경 후 다른 Pain 영역 탐색 |
| 응답이 매우 짧음 (1~2문장) | "조금 더 구체적으로 말씀해 주시겠어요? 그때 상황이..." 추가 |
| Yes-man 감지 (3회 연속 긍정) | 비판적 반례 질문 삽입: "그렇지 않다고 느낀 순간은 없으셨나요?" |
| 주제 이탈 | 부드럽게 재집중: "흥미롭네요. 그럼 {원래 주제}로 다시 돌아가서..." |

### 2-B. 탐침 질문 (Probing)

응답에서 의미 있는 신호를 감지하면 추가 탐침한다:

| 신호 | 탐침 질문 |
|------|----------|
| "그냥" 사용 ("그냥 참아요") | "그냥 참으신다고 하셨는데, 그 순간 어떤 기분이셨어요?" |
| 대체 행동 언급 | "그 방법을 쓰게 된 이유가 있으셨나요? 처음엔 어떻게 알게 되셨어요?" |
| 타인 언급 | "그분들도 비슷한 불편함을 겪고 계신가요?" |
| 포기 언급 | "포기하셨을 때 어떤 생각이 드셨어요?" |
| 가격 언급 | "구체적으로 얼마 정도면 합리적이라고 느끼실 것 같으세요?" |

## 3. 인터뷰 품질 검증 체크리스트

인터뷰 완료 후 아래 항목을 점검한다. 실패 항목이 있으면 해당 단계 재실행.

| 검증 항목 | 기준 | 실패 시 처리 |
|---------|------|-----------|
| 총 질문 수 | 8~10개 | 부족하면 추가 탐침 |
| Problem Probe 응답 길이 | 평균 3문장 이상 | "더 구체적으로" 재질문 |
| Yes-man 비율 | 긍정 응답 ≤ 60% | 비판적 질문 재삽입 |
| 구체적 경험 언급 수 | 최소 2건 | Probe 추가 |
| 새로운 Pain 발견 수 | 최소 1건 | 탐색 범위 확장 |
| 메타데이터 완성도 | 4개 항목 모두 작성 | 누락 항목 보완 |

## 4. 페르소나별 어조 가이드

### 20대 (tech-savvy, 가격 민감)
```
- 짧고 임팩트 있는 문장
- "솔직히", "진짜", "그니까요" 같은 구어체 자연스럽게 포함
- 모바일 우선 경험 ("폰으로 보다가", "앱에서")
- 가격보다 경험/편리함 중시 경향
```

### 30대 직장인 (시간 희소, 효율 추구)
```
- 업무 컨텍스트 연결 ("퇴근하고 나서", "점심때 틈틈이")
- "귀찮아서", "바쁘니까" 표현 빈도 높음
- 비교·검색 행동 패턴 ("다른 거 찾아보다가")
- 가성비 언급 빈도 높음
```

### 40대 이상 (낮은 기술 친숙도)
```
- "잘 모르겠는데요", "복잡하면..." 보류 표현 자주 사용
- 기존 방식에 대한 강한 애착 ("하던 대로가 편해서")
- 신뢰 신호 중시 ("아는 사람이 써보래서")
- 구체적 학습 곡선 언급
```
