## 1. Data Catalog 성숙도 모델

단순 "이벤트·속성 목록 뷰어"가 아닌, 아래 5개 계층을 순서대로 쌓아가는 구조.

```
Level 5. Intelligence   ← LLM 탐색, 자동 메타데이터 추론, 임팩트 분석
Level 4. Governance     ← 소유권, 변경 관리, 감사 로그
Level 3. Lineage        ← 출처 추적, 다운스트림 활용 파악
Level 2. Context        ← 정의, 예시값, 태그, 오너십
Level 1. Inventory      ← 존재하는 데이터 자산의 목록화
```

| 레벨 | 핵심 질문 | MATCH 현재 위치 | Phase 목표 |
|------|---------|--------------|-----------|
| Level 1 | 무엇이 존재하는가? | 미완성 (Properties 뷰어 수준) | Phase 1 완성 |
| Level 2 | 이 데이터가 무엇을 의미하는가? | 없음 | Phase 1 진입 |
| Level 3 | 어디서 와서 어디에 쓰이는가? | 없음 | Phase 2 진입 |
| Level 4 | 누가 어떻게 관리하는가? | 없음 | Phase 2~3 |
| Level 5 | 시스템이 스스로 돕는가? | 없음 | Phase 3 |

## 4. 핵심 시사점

### Amplitude를 Source of Truth로 활용하는 전략

Level 1~2 일부는 Amplitude가 이미 잘 커버함.
MATCH에서 동일 기능을 재구현하면 관리 포인트만 늘어나는 중복 투자 위험.

**권장 구조:**
```
Amplitude (이벤트·속성 스키마 원본 관리)
        ↓ 일 1회 동기화
MATCH Data Explorer (MATCH 맥락 메타데이터 레이어)
        ↓ 연결
Braze 세그먼트 / Activation / 개인화 변수
```

### MATCH Data Explorer의 차별점

Amplitude는 "Amplitude 안에서의 사용"만 추적하지만,
MATCH Data Explorer는 **Braze 세그먼트·Activation·개인화 변수까지 연결**해야 함.
이 활용 맥락 연결(E2-4)이 Amplitude Catalog보다 마케터에게 더 유용한 핵심 이유.

### Phase 1이 전체 로드맵 품질을 결정하는 이유

- E5-1 LLM 탐색은 E2-1 메타데이터 완성도가 선행되어야 품질 보장
- E1-2 다중 데이터 유형 지원은 Phase 1의 IA가 Generic Schema를 수용할 수 있게 설계되어야 함
- Phase 1에서 구조적 결함이 생기면 Phase 2~3에서 재설계 강제
