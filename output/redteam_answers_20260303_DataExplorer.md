## A. 문제 정의

### A-01. 데이터 탐색 리소스 정량화

**상황 설정**: 마케터가 '아우터 카테고리 조회 후 미구매' 세그먼트를 Braze에서 만들려는 상황.

| 단계 | 소요시간 | 블로커 |
|------|---------|--------|
| Braze에서 이벤트 탐색 (설명 없음, 유사 이름 5~6개) | 10~15분 | 이벤트 설명 없음 |
| 텍소노미 명세서 탐색 (최신 여부 불명확) | 20~30분 | 버전 신뢰 불가 |
| 실제 속성값 확인 (DA Slack 요청 + 대기) | 1시간~1일 | 사람 의존 |
| 값 적용 후 검증 (형식 불일치로 세그먼트 0명) | 30분~1시간 | 값 형식 불일치 |
| **합계** | **2시간~2일** | |

**PRD/2-Pager 반영 제안**
- 2-Pager `배경/문제` 섹션: "마케터 1인이 새로운 속성을 처음 활용하기까지 평균 4~8시간 소요 (DA 대기 포함 시 1~2일). 팀 내 마케터 N명 × 주 X회 기준 월 YY시간 기회비용."
- PRD `성공 지표` 섹션: `속성 활용 준비 시간 / 현재 4~8시간 → 목표 1시간 이내`로 Before/After 기준 수치로 전환.
- 산출 로직 원본은 2-Pager에만 두고, PRD는 결론 수치 + 2-Pager 링크.

### A-02. "메타데이터 없음 → 잘못된 데이터 활용" — 비약인가?

**결론: 비약 아님. 단, 중간 메커니즘이 생략되어 있어 설득력이 약하다.**

현재 PRD 문장의 문제:
```
메타데이터 없음 → (생략) → 잘못된 데이터 활용
```

**수정 제안**:
> "속성의 정의·허용 값·수집 상태 정보가 없으면, 마케터는 유사한 이름의 이벤트 중 어느 것이 맞는지 알 수 없어 추측으로 속성을 선택하거나 DA에게 의존하게 된다. 추측으로 선택한 속성이 실제와 다를 경우, 잘못된 사용자 세그먼트로 캠페인이 발송되거나 개인화 변수가 오작동하는 결과로 이어진다."

**추가 권고**: 실제 발생 사례(잘못된 세그먼트 타겟팅, DA 문의 건수 등) 최소 1건을 병기하면 논리 완결.

### A-03. 마케터 vs 운영자 vs 엔지니어 — 탐색 방식 및 필요 정보 차이

| 구분 | 핵심 질문 | 탐색 방식 | 필요한 정보 |
|------|---------|---------|-----------|
| **마케터** | "이 속성 써도 되나? 값이 뭐야?" | 의미·목적 중심 브라우징 | 한국어 설명, 예시값, 마지막 수집일, 활용 현황 |
| **운영자** | "이 데이터 지금 잘 들어오고 있나?" | 이상 감지 중심 (수집 현황 → 드롭 감지) | 수집 추이 그래프, 누락 알림, 히스토리 |
| **엔지니어** | "스키마가 뭐야? 타입이 뭐야?" | 스키마 레퍼런스 탐색 | 데이터 타입, 원천 시스템, Nullable 여부 |

**Phase 1 = 마케터 1순위 설계 함의**:
- 강조: 한국어 설명, 예시값, 활용 현황 (어느 캠페인/코호트에서 쓰이는지)
- 축소 가능: 스키마 타입 상세, 원천 시스템 경로 (엔지니어용)
- 주의: 수집 현황 숫자(운영자용)를 마케터에게 그대로 노출하면 혼란. **"정상 수집 중 ✅" 처럼 해석된 상태값으로 노출** 권장.

### A-05. 예측된 미래 수요 — 어디에 어떻게 반영할까

**2-Pager `배경/문제` 섹션 수정 제안**:
```markdown
현재 MATCH에서 유저 데이터 외 캠페인 소재·유저 통계 데이터를
탐색하려는 직접적 수요는 측정되지 않았다.

아래 간접 신호를 근거로 수요가 존재할 것으로 가정한다:
- 마케터 인터뷰 중 "소재별 퍼포먼스를 유저 속성과 같이 보고 싶다" 언급 N건
- 타 플랫폼(Amplitude, Mixpanel)에서 multi-entity 탐색이 표준화됨

→ Phase 1은 이 가설을 검증하는 실험이다.
   캠페인 소재 탐색 기능의 실제 사용률을 출시 후 8주 내 측정한다.
```

**PRD `성공 지표` 섹션에 가설 검증 지표 추가**:

| 가설 | 검증 지표 | 기준 | 판단 시점 |
|------|---------|------|---------|
| 마케터가 유저 속성 외 데이터도 탐색한다 | 캠페인 소재 탭 UV | 주간 MAU의 20%↑ | 출시 후 8주 |
| 미충족 수요가 실재한다 | 탐색 후 실제 활용률 | UV 대비 30%↑ | 출시 후 8주 |

기준 미달 시: Phase 2 캠페인 소재 고도화 범위 재검토.

### A-06. LLM 범위 축소 미공유 — 어떻게 추가할까

**PRD에 `범위 결정 로그 (Scope Decisions)` 섹션 신설 제안**:

```markdown
## 범위 결정 및 제외 사항

### [SD-01] 자연어·LLM 기반 탐색 — Phase 1 제외

- 2-Pager 원문: "향후 자연어·LLM 기반 인터랙션으로 진화"
- Phase 1 결정: 제외
- 결정 근거:
  1. 속성 메타데이터 자체가 불완전하여 LLM이 참조할 컨텍스트 부재
  2. 기본 탐색 UX 검증 없이 LLM 도입 시 사용성 문제 인과 파악 불가
  3. 개발 복잡도 대비 Phase 1 목표 달성에 불필요
- Phase 1이 Phase 2를 막지 않는 이유:
  - 속성 메타데이터 구조를 LLM RAG 인덱싱에 호환 가능하게 설계
  - IA 구조 재설계 없이 검색창 UX 추가 가능
- 공유 대상 및 방법: {PM 작성 필요 — 팀장/기획자 대상 공유 일정 명시}
- 결정일: 2026-03-XX
```

이 섹션이 있으면 "LLM은 언제 하는 거예요?" 질문에 문서 링크로 대응 가능.

### B-01. OKR 수치 설정 제안

**Step 1 — 출시 전 2~4주 내 베이스라인 먼저 측정**

| 측정 항목 | 방법 |
|---------|------|
| DA에게 오는 속성 관련 Slack 문의 건수 | 채널 수동 카운트 또는 태그 |
| 기존 Properties 메뉴 주간 UV | MATCH 내부 로그 |
| 마케터가 새 속성 활용까지 걸리는 시간 | 인터뷰 2~3명 |

**Step 2 — OKR 초안**

```
Objective: 마케터가 데이터를 스스로 탐색하고 신뢰할 수 있는 환경을 만든다.

KR1. 마케터의 속성 관련 DA 문의 40% 감소 (베이스라인: 출시 전 측정)
KR2. Data Explorer 주간 활성 사용자 = 마케터 팀 MAU의 60%
KR3. 속성 활용 준비 시간 1시간 이내 (현재 추정: 4~8시간)
```

**수치 확정이 어려울 때 PRD 문구**:
```markdown
KR 수치는 출시 전 2주 내 베이스라인 측정 후 확정.
현재 목표 범위: DA 문의 30~50% 감소 / 마케터 MAU 50~70% 도달.
확정일: 2026-03-XX | Owner: {PM 이름}
```

## 핵심 가설 (검증 대상)

가설: "속성 정의·예시값·수집 상태 정보가 제공되면,
      마케터의 잘못된 속성 선택 빈도가 줄어든다."

### B-04. Adoption 지표 설정

Adoption을 단일 숫자로 보지 않고 단계별로 설정:

```
Awareness → Activation → Retention → Depth
```

| 단계 | 지표 | 목표 | 측정 방법 |
|------|------|------|---------|
| **Awareness** | Data Explorer 메뉴 클릭률 (전체 로그인 대비) | 40% | MATCH 로그 |
| **Activation** | 진입 후 속성 상세 패널 클릭 비율 | Awareness의 50% | 클릭 이벤트 |
| **Retention** | 4주 연속 1회 이상 방문한 마케터 비율 | Activation의 60% | 코호트 분석 |
| **Depth** | 탐색 후 Braze/코호트 실제 적용 비율 | Activation의 30% | DA 확인 필요 |

**Properties → Data Explorer 전환율 별도 추적**:

| 전환 지표 | 목표 | 시점 |
|---------|------|------|
| 기존 Properties UV 감소율 | 50% 이하 | 출시 8주 후 |
| Data Explorer UV / (Data Explorer + Properties UV) | 70% 이상 | 출시 8주 후 |

Properties가 병존하는 동안 두 메뉴의 UV 비율이 가장 직관적인 전환 지표.
