## 결론

> MATCH는 Braze를 대체하지 않는다.
> ML 모델은 이미 운영 중이다. 크로스플랫폼 데이터도 이미 있다.
> **이 자산들이 CRM 타겟팅에 연결되지 않아 매일 낭비가 발생하고 있다.**
> MATCH는 이미 존재하는 것들을 연결하는 마지막 조각이다.

Braze에 연간 10~14억을 내고 있다. 그 Braze가 사용하는 오디언스는
무신사가 이미 만들어 놓은 ML 모델 결과를 전혀 반영하지 못한다.
이것이 현재 가장 큰 비효율이다.

첫 번째 질문은 "왜 직접 짜냐"가 아니라 **"왜 이미 만든 모델을 아직도 안 쓰고 있냐"** 다.

## 2. Braze의 구조적 한계 — 발송 채널의 본질

Braze는 마케팅 자동화 발송 플랫폼이다. 이 정의가 곧 한계다.

### Braze가 할 수 없는 것

**① 오디언스를 스스로 만들지 못한다**

Braze가 생성하는 세그먼트는 Braze 안으로 들어온 이벤트 데이터 기반이다.
무신사의 스타일 그래프, 29CM 콘텐츠 취향 임베딩, 추천 클릭 이력 — 이 데이터들은
Braze가 수집하지 않거나, Currents(유료 Add-On)로 내보내야 역산 가능하다.
즉, 우리가 가진 가장 정교한 유저 이해가 Braze 오디언스에는 반영되지 않는다.

**② 도메인 간 신호를 연결하지 못한다**

```
추천 클릭했지만 미구매     → CRM으로 연결? → Braze는 모름
광고 타겟팅 받은 유저     → CRM 중복 노출 제거? → Braze는 모름
CRM 수신 거부한 유저      → 광고 exclusion? → Braze는 모름
```

각 도메인은 각자 최적화한다. 결과적으로 동일 유저가 하루에 광고·추천·CRM을
각자의 로직으로 접촉하며, 서로를 모른다.

**③ 데이터를 잠근다**

Currents(데이터 export)가 유료 Add-On이라는 구조는,
Braze 안의 발송 결과 데이터를 꺼내 분석하려면 비용을 더 내야 한다는 의미다.
이는 Braze가 데이터 흐름의 게이트키퍼 역할을 하고 있음을 뜻한다.

## 3. "SaaS 조합으로 해결되지 않는가?" — 반론 검토

이 질문은 유효하다. 솔직하게 검토한다.

| 주장 | 실제 가능 여부 | 하지만 |
|------|-------------|-------|
| Databricks → Hightouch → Braze로 임베딩 오디언스 sync | ✅ 기술적으로 가능 | 4개 팀·4개 계약·4개 데이터 모델 동기화가 필요. 실제 운영 마찰비용이 발생 |
| Amplitude Computed Traits로 고도 세그먼트 생성 | ✅ 가능 | 무신사 내부 ML 모델(추천·임베딩) 결과를 직접 주입할 수 없음 |
| Braze Connected Content로 실시간 추론 | ✅ 가능 | 발송 건마다 API 호출 — 대규모 발송 시 latency와 비용 이슈 |
| 서비스 간 연결(iPaaS/CDP)로 통합 | ✅ 가능 | 연결 레이어가 늘수록 추가 계약·비용·운영 복잡도 증가 |

**핵심 결론:** SaaS 조합으로 "기술적으로" 가능하지 않은 것은 없다.
문제는 **그 조합의 총비용(TCO)이 얼마인가**와,
**무신사의 고유 데이터 자산을 외부에 맡기는 전략적 리스크**다.

## 4. MATCH가 담당하는 레이어

MATCH는 MarTech 스택에서 현재 공백인 레이어를 채운다.

```
[데이터 레이어 — 무신사 고유 자산]
  온라인 행동 이력 / 스타일 그래프 / ML 모델 결과
  + 오프라인 구매 / PLCC 소비 패턴 / 크로스플랫폼 통합 뷰
        ↓
[MATCH — 오디언스 인텔리전스 레이어]  ← 현재 공백
  • 내부 ML 결과 → 오디언스 세그먼트 변환
  • 광고·추천·CRM 오디언스 동기화
  • 도메인 간 중복·순서 제어
        ↓
[실행 레이어]  Braze(CRM) / 광고 플랫폼 / 추천 시스템
```

Braze는 계속 Braze다. MATCH는 Braze 위·앞단에 위치하며,
지금 Braze가 처리하는 오디언스 생성 로직의 품질을 높인다.

### MATCH가 만드는 오디언스의 차별점

| 현재 Braze 세그먼트 | MATCH 기반 세그먼트 |
|------------------|------------------|
| Braze에 들어온 이벤트 기반 | 내부 추천 모델·임베딩 결과 직접 반영 |
| CRM 단독 최적화 | 광고 exposure 제외, 추천 클릭 포함 |
| 수동 조건 설정 | 모델 기반 자동 갱신 |
| 채널별 독립 운영 | 채널 간 노출 조율 가능 |
| 온라인 행동만 반영 | 오프라인·PLCC·크로스플랫폼 통합 뷰 반영 |

## 4.5 무신사 고유 데이터 자산 — 외부 SaaS가 접근할 수 없는 신호들

이 섹션이 MATCH 구축 논거의 핵심이다.
기술적 가능성이 아니라 **데이터 독점성**이 차별점이다.

### ① ML 기반 속성 — 가장 강력한 논거

**ML 모델은 이미 운영 중이다. 그 결과가 CRM 타겟팅에 반영되지 않고 있다.**

Braze가 만드는 세그먼트는 이벤트 로그 기반이다.
"최근 7일 내 상품 조회" 같은 조건이다.
현재 내부에서 운영 중인 ML 모델들은 이보다 훨씬 정교한 결과를 이미 생산하고 있다.

| 속성 유형 | 예시 | 모델 운영 현황 | Braze 반영 여부 |
|---------|-----|-------------|--------------|
| LTV 예측 스코어 | 향후 6개월 구매 예상 금액 상위 10% | ✅ 운영 중 | ❌ 미반영 |
| 이탈 위험 스코어 | 30일 내 이탈 확률 70% 이상 | ✅ 운영 중 | ❌ 미반영 |
| 스타일 취향 임베딩 | 미니멀 스트리트 + 가격 민감 유저 | ✅ 운영 중 | ❌ 미반영 |
| 브랜드 친화도 벡터 | A 브랜드 이탈 → B 브랜드 전환 예측 | ✅ 운영 중 | ❌ 미반영 |
| 구매 시점 예측 | 다음 구매 가능성이 가장 높은 날짜 | ✅ 운영 중 | ❌ 미반영 |

> **이것이 현재 가장 큰 낭비다.**
> ML 모델 개발·운영 비용은 이미 집행되고 있다.
> 그 결과가 CRM 오디언스에 연결되지 않아, 매 캠페인마다 Braze의 이벤트 기반 세그먼트를 수동으로 만들고 있다.
> MATCH는 이미 존재하는 모델 결과를 Braze로 자동 연결하는 마지막 단계다.

**"Braze Custom Attribute에 모델 값을 넣으면 되지 않나?"**

가능하다. 하지만 모델이 재학습될 때마다 수천만 유저의 속성값을 자동으로 최신화하는
파이프라인이 없으면 무의미하다. 수동으로 업데이트하는 순간 모델 결과는 이미 stale이다.
그 자동화 파이프라인이 MATCH의 핵심 기능이다.

### ② 크로스플랫폼 통합 뷰

현재 Braze 계약은 무신사·29CM·무신사글로벌·솔드아웃을 개별 인스턴스로 운영한다.
(SLA 명시: "서비스명: Musinsa, Musinsaglobal, Soldout, 29cm")

이 구조에서는 동일 유저가 4개 플랫폼에서 어떻게 행동하는지를
**Braze 안에서 통합해서 보는 것이 불가능하다.**

```
무신사에서 남성 스트리트 구매  ┐
29CM에서 라이프스타일 콘텐츠 탐색 ├→ Braze: 각각의 독립 프로필
솔드아웃에서 한정판 즐겨찾기 ┘     플랫폼 간 연결 없음

                               ↓ MATCH

통합 뷰: "한정판 관심 + 라이프스타일 취향 + 스트리트 구매력"
→ 세 가지 신호를 결합한 오디언스 생성 가능
```

이 통합 뷰는 어떤 외부 SaaS도 계약 구조상 자동으로 제공할 수 없다.

### ③ 오프라인 구매 데이터

무신사 오프라인 스토어(무신사 스탠다드 등) 구매 이력은
온라인 행동 데이터와 결합할 때 두 가지 오디언스를 새롭게 만든다.

| 오디언스 유형 | 설명 | 현재 CRM 반영 여부 |
|------------|-----|-----------------|
| 온라인 이탈 + 오프라인 활성 | 앱 미접속 30일 이상이지만 오프라인 구매 활성 유저 | ❌ 미반영 |
| 오프라인 전용 구매자 | 온라인 행동 거의 없지만 오프라인 고가 구매 반복 | ❌ 미반영 |
| O2O 전환 트리거 유저 | 온라인 탐색 → 오프라인 구매 패턴 보유 | ❌ 미반영 |

### ④ PLCC 소비 데이터 (조건부)

무신사 현대카드(PLCC) 보유자의 카드 소비 패턴은
**무신사 앱 내 행동으로 포착되지 않는 소비 전체**를 포함한다.

| 활용 가능 신호 | 의미 | 오디언스 적용 예시 |
|-------------|-----|----------------|
| 패션 카테고리 총 지출 | 무신사 외 패션 구매 규모 | 경쟁사 이탈 유저 재타겟팅 |
| 생활비 대비 패션 지출 비율 | 패션 관여도 추정 | 고관여 유저 프리미엄 세그먼트 |
| 무신사 구매 비중 | 무신사 share of wallet | SOW 낮은 고잠재 유저 집중 공략 |

> **⚠️ 중요 조건**: PLCC 데이터 활용은 개인정보 수집·이용 동의 범위,
> 카드사와의 데이터 제공 계약 범위 내에서만 가능하다.
> 법무·컴플라이언스 검토 없이 타겟팅에 직접 적용할 수 없다.
> 단, 동의 범위가 확보된다면 경쟁사 대비 구조적으로 우위에 있는 신호다.

## 5. 증분 가치 계산

### 레버 1: 오디언스 품질 개선 → CTR 향상

| 지표 | 현재 | 목표 | 근거 |
|------|------|------|------|
| 개인화 캠페인 CTR | __________% | +__________% | 내부 A/B 실험 필요 |
| CTR 향상으로 인한 추가 GMV | | __________ 억원/년 | CTR × 전환율 × AOV |

> **이 수치가 없으면 어떤 ROI 주장도 성립하지 않는다.**
> Phase 1 착수 전 CRM 성과 Baseline 확보가 선행 조건이다.

### 레버 2: Currents Add-On 비용 내재화

MATCH가 오디언스 로직을 내부에서 처리하면,
발송 결과 데이터를 Braze 밖으로 꺼낼 필요가 줄어든다.

| 항목 | 현재 | MATCH 후 |
|------|------|---------|
| Currents 연간 비용 | $32,061~$43,720 (약 4,400만~6,000만원) | 일부 절감 가능 |

### 레버 3: MAU 과금 구조 완충

Braze는 MAU가 늘수록 비용이 선형 증가한다.
MATCH가 타겟 오디언스를 정교화하면, 동일 MAU에서 효율을 높여
**MAU 증가 없이 성과를 높이는 방향**이 가능하다.

| 시나리오 | 무신사 Braze 3년차 비용 |
|---------|---------------------|
| 현재 계획대로 | $1,004,667 (약 13.9억) |
| MAU 초과 발생 시 | $0.041238/MAU 추가 — 100만 명 초과 시 연간 +약 6.8억 |

### 레버 5: 고유 데이터 자산 활용 — 현재 사용하지 못하는 신호

이 레버는 기존 오디언스 "개선"이 아니라 **현재 완전히 누락된 오디언스 발굴**이다.

**ML 속성 활용 (LTV·이탈 예측) — 모델 이미 운영 중, 연결만 남은 상태:**

현재 CRM은 행동 기반 세그먼트만 사용한다.
ML 모델은 이미 더 정교한 결과를 생산하고 있지만 CRM에 전달되지 않는다.
연결되는 순간 다음 두 가지가 즉시 가능해진다:
- 고LTV 예측 유저 집중 공략 → 동일 발송 비용으로 GMV 효율 향상
- 이탈 임박 유저 사전 개입 → 이탈율 감소 → LTV 방어

| 시나리오 | 가설 | 정량화 |
|---------|-----|-------|
| 고LTV 예측 상위 10% 집중 발송 시 CTR 변화 | 무관심 유저 제외 → 전체 CTR 상승 | A/B 실험 설계 — __________ |
| 이탈 임박 유저 인터벤션 캠페인 리텐션율 | 이탈 방어 유저당 LTV × 이탈율 감소분 | __________ |

> 이 두 실험은 MATCH Phase 1 연결 완료 직후 즉시 시작 가능하다. 모델이 이미 있기 때문이다.

**크로스플랫폼 미접점 유저:**

4개 플랫폼 통합 시 "한 플랫폼에서만 비활성이나 다른 플랫폼에서 활성인 유저"가 드러난다.
이 유저는 현재 Braze에서 이탈 유저로 분류되어 발송 제외 상태다.

| 오디언스 | 현재 처리 | MATCH 후 |
|---------|---------|---------|
| 무신사 비활성 + 29CM 활성 유저 | CRM 대상 제외 | 플랫폼 교차 리인게이지 가능 |
| 무신사 비활성 + 오프라인 활성 유저 | 이탈 유저로 분류 | 오프라인 행동 기반 온라인 복귀 캠페인 |

이 유저군의 규모: __________ (크로스플랫폼 데이터 집계 필요)

**PLCC 기반 Share-of-Wallet 타겟팅 (법적 범위 내 확보 전제):**

무신사 SOW(Share of Wallet)가 낮지만 패션 총지출이 높은 유저 =
**경쟁사에 돈을 쓰고 있는 고잠재 고객**이다. 이 오디언스는 현재 어떤 SaaS도 만들 수 없다.

## 6. 투자 의사결정 프레임

### Build vs Augment Matrix

```
                  낮은 전략 가치          높은 전략 가치
                 ─────────────────────────────────────
SaaS로 충분       │  Buy & Forget      │  Partner       │
                 │  (ex: Slack, Zoom) │  (ex: Braze)   │
                 ├────────────────────┼────────────────┤
SaaS로 불충분     │  Delay             │  ★ BUILD       │
                 │                   │  (MATCH 위치)  │
                 └────────────────────┴────────────────┘
```

MATCH는 "전략 가치 高 + SaaS 불충분" 영역에 위치한다.
단, 이 판단은 **오디언스 인텔리전스 레이어에 한정**된다.
Braze 자체(발송 채널)는 "Partner" 영역 — 계속 사용한다.

### TCO 비교 (3년 기준)

| 시나리오 | 비용 항목 | 3년 추정 |
|---------|---------|---------|
| **현상 유지** (SaaS 조합 추가) | Braze $2.26M + Hightouch/CDP 추가 + 통합 엔지니어링 인건비 | $2.26M + __________ |
| **MATCH 구축** | Braze $2.26M 유지 + MATCH 개발·운영 | $2.26M + __________ |
| **예상 절감·증분** | Currents 일부 대체 + 오디언스 품질 개선 GMV | __________ |

> 핵심 질문: "SaaS 조합 추가 비용" vs "MATCH 구축 비용" 중 무엇이 더 큰가?
> 이 계산이 완성되어야 의사결정이 가능하다.

## 7. 단계적 접근 — 리스크를 최소화하는 경로

Big Bang 구축이 아니라, **CRM에서 증명 → 확장** 순서로 진행한다.

| Phase | 기간 | 범위 | 성공 기준 |
|-------|------|------|---------|
| **Phase 0 (현재)** | ~2026 Q1 | CRM 오디언스 일부 구축 완료 | 베이스라인 측정 체계 확보 |
| **Phase 1** | 2026 Q2 | CRM 오디언스 품질 고도화 + 성과 A/B 비교 | CTR __________ % 향상 실증 |
| **Phase 2** | 2026 Q3~Q4 | 광고 오디언스 연결 (중복 제거, exclusion) | 광고 효율 __________ % 개선 |
| **Phase 3** | 2027 | 추천 오디언스 연결 + 멀티도메인 최적화 | 도메인 통합 GMV 증분 측정 |

**Phase 1에서 ROI가 증명되지 않으면 Phase 2로 가지 않는다.**
C레벨 승인은 Phase 0→1 전환점과 Phase 1→2 전환점, 두 번으로 나눈다.
