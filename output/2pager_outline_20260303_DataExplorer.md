## 2. Problem Definition

### Context ✅ → 그대로 사용 가능

- MATCH 데이터 자산이 유저 데이터 중심에서 캠페인·카탈로그로 확장 중
- 기존 CDP 프레임 기반 구조가 다양한 데이터 유형을 포괄하기 어려운 구조
- 데이터는 축적되고 있으나, 탐색과 의사결정에 소요되는 비용은 줄지 않고 있음

### Pain Point ✅ → 두 덩어리로 구조화

**Pain 1. 탐색 경험 공백**
- 데이터 유형이 다양해졌는데 유저 데이터 중심 탐색 구조만 남아 있음
- 마케터·운영자·엔지니어가 각자 다른 탐색 목적을 가지지만 단일 뷰만 제공

**Pain 2. 메타데이터 공백**
- 이벤트·속성의 수집 상태·정의·활용 맥락을 알 수 없음
- 마케터가 DA에게 의존하거나 추측으로 데이터 선택 → 잘못된 세그먼트 타겟팅·개인화 오류 발생

### Impact 🔶 → 수치 보강 필요

기존 문서에 있는 내용:
- 마케터가 새로운 속성 활용까지 **평균 4시간 리드타임** 소요
  - 이벤트 탐색(15분) + 명세서 탐색(30분) + DA 요청 대기(~1일) + 값 검증(1시간)
- 팀 내 마케터 **N명 × 주 X회 × 2시간 = YY시간** 기회비용 (N, X, YY 공란 — 확정 필요)

**채워야 할 것:**
- 실제 마케터 수, 주간 신규 속성 활용 빈도 수치 확정
- 실제 발생한 잘못된 세그먼트 타겟팅 사례 최소 1건 병기
- 없으면 `[추정값]` 명시 후 진행

## 3. Proposed Solution

### Solution Overview ✅ → 1~2문장으로 압축

> "MATCH 내 모든 데이터 자산(로그·유저 통계·카탈로그·캠페인)을 공통 텍소노미로 일원화하고,
> 메타데이터·품질 정보를 제공하는 데이터 디스커버리 플랫폼."

### Key Approach & Concept 🔶 → 선택 논리 추가 필요

기존 문서는 "무엇을 하겠다"는 목표 나열. "왜 이 방향인가"가 빠져 있음.

**추가할 내용 (2~3줄):**
> "단순 검색 뷰어가 아닌 Data Catalog 레이어로 설계하는 이유는,
> 마케터의 탐색 효율화(Phase 1)를 넘어 데이터 거버넌스(Phase 2~3)까지 연결하는
> 단일 진실의 원천(Single Source of Truth)을 확보하기 위함이다.
> LLM 기반 탐색은 메타데이터 완성도가 선행되어야 품질이 보장되므로,
> Phase 1의 구조적 완성도가 전체 로드맵의 품질을 결정한다."

### Phased Approach ❌ → 신규 작성

| 구분 | Phase 1 | Phase 2 | Phase 3 |
|------|---------|---------|---------|
| **핵심 목표** | 신뢰할 수 있는 카탈로그 기반 구축 | 데이터 유형 확장 + 품질 관리 | 의도 기반 탐색 + 거버넌스 완성 |
| **주요 사용자** | 마케터 | 마케터 + 운영자 | 전체 (마케터·운영자·엔지니어) |
| **Goals** | DA 없이 마케터가 속성 자율 탐색 | 캠페인·통계 데이터까지 탐색 범위 확장 | 자연어로 누구나 데이터 탐색 |
| **Key Metrics** | 속성 활용 준비 시간 1시간 이내 / DA 문의 40% 감소 | 신규 데이터 유형 탐색 UV / 스키마 자동화율 | 자연어 탐색 이용률 / 메타데이터 완성도 |
| **Milestone** | 26년 1Q | 26년 3Q (안) | 27년 (안) |
| **Scope** | 통합 카탈로그 뷰 / 메타데이터 패널 / 수집 현황 / 활용 맥락 | 캠페인 소재 통합 / Generic Schema / Stewardship 워크플로우 / 품질 대시보드 | LLM 검색 / 리니지 시각화 / 자동 메타데이터 / 임팩트 분석 |

### Assumptions & Risks ❌ → 신규 작성

| 구분 | 내용 |
|------|------|
| Assumption | 마케터가 메타데이터만 있어도 DA 없이 속성을 선택할 수 있다 |
| Assumption | 캠페인 소재 탐색 수요가 실재한다 (A-05: 예측 수요 — Phase 1에서 검증 예정) |
| Risk | 메타데이터 입력 담당자(Data Steward) 미지정 시 빈 카탈로그로 출시 가능 |
| Risk | Properties → Data Explorer 전환 시 기존 사용자 혼란 (리다이렉트 계획 필요) |
| Risk | LLM 도입 전제로 설계하지 않으면 Phase 3에서 IA 재설계 필요 |
