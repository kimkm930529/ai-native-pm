Concept

우리의 목표는 사용자가 데이터가 어디 있는지를 넘어, 그 의미를 정확히 이해하고 신뢰하며, 비즈니스에 활용할 수 있는 '데이터 익스플로러'를 구축하는 것입니다. 우리 제품 방향성은 AI 기반 검색과 거버넌스된 메타데이터 컨텍스트를 Data Explorer 내에 통합하여, 관리와 탐색 사이의 경계를 완전히 허무는 것입니다.

메타데이터 플랫폼의 유형

이를 위해 혼재되어 사용되는 유관 개념들을 명확히 구분하고, 각 도구가 해결하고자 하는 핵심 페인 포인트(Pain Point)를 비교 분석합니다.

Data Explorer의 성숙도: 5단계 의존성

Level 구조의 배경

결론부터 말하면, 업계 공통의 Data Catalog 성숙도 모델을 기반으로 하되 MATCH 맥락에 맞게 재구성한 것입니다.

1. 이론적 배경: 어디서 온 프레임인가

업계에서 Data Catalog의 진화를 설명할 때 반복적으로 등장하는 구조입니다.

동일 순서로 단계적 구축 사례 공개

표현은 조금씩 달라도 순서는 모두 동일합니다. 이 순서에는 이유가 있습니다.

2. 핵심 논리: 왜 이 순서여야 하는가

각 Level은 이전 Level이 없으면 물리적으로 불가능합니다.

즉, 5개 Level은 임의로 나눈 것이 아니라 의존성 체인입니다.

3. 실패 사례가 이 순서를 증명함

잘못된 시도

결과

L1 없이 L2 시작 (설명부터 쓰기)

카탈로그에 없는 속성에 설명이 붙어 있거나, 새 속성이 추가돼도 감지 못 함

L3 없이 L4 시작 (거버넌스부터)

속성을 비활성화했는데 어떤 캠페인이 영향받는지 모름 → 거버넌스 공허해짐

L2 없이 L5 시작 (LLM 탐색부터)

"구매 관련 속성 보여줘" → 설명이 없으니 이벤트명만 보고 추측 → 부정확한 결과

MATCH의 경우 LLM 탐색을 Phase 3로 뒤로 뺀 이유가 바로 이것입니다. L2 메타데이터 완성 없이 L5를 올리면 품질을 보장할 수 없습니다.

4. MATCH에서 이 순서가 특히 중요한 이유

일반적인 Data Catalog과 달리 MATCH는 탐색 결과가 실제 마케팅 실행으로 연결됩니다.

실행 오류의 비즈니스 임팩트가 크기 때문에, 기반을 건너뛰는 것이 더 위험합니다.

한 줄 요약

Level 순서는 "다음 단계는 반드시 이전 단계에 의존한다"는 기술적 의존성에서 나온 것이며, 업계 공통 Data Catalog 성숙도 프레임워크와 일치합니다. 순서를 뒤집으면 각 단계의 품질이 보장되지 않습니다.

Airbnb

배경 및 당면 과제

  • 에어비앤비는 2008년 설립 이후 폭발적인 성장을 거듭하며 데이터의 양과 복잡성이 기하급수적으로 증가 → 이 과정에서 PM과 데이터 인력들은 다음과 같은 '데이터 병목 현상'에 직면

    • 데이터 파편화(Siloed Data): 데이터가 각 팀과 도구별로 고립되어 전사적인 '단일 진실 공급원(Single Source of Truth)' 유지가 불가능해졌습니다.

    • 구전 지식(Tribal Knowledge)의 한계: 특정 데이터를 어떻게 찾고 사용하는지에 대한 정보가 일부 전문가의 머릿속에만 머물러 있어, 비전문가나 신규 입사자는 데이터에 접근조차 하기 어려웠습니다.

    • 신뢰의 결여: 메타데이터와 맥락(Context)이 부족하여 사용자가 찾은 데이터가 최신인지, 정확한지 판단할 수 없었고 이는 데이터 활용을 포기하게 만들었습니다.

해결책: Dataportal의 탄생

  • 에어비앤비는 단순한 저장소를 넘어, 모든 직원이 데이터를 탐색(Explore), 발견(Discover), 신뢰(Trust)할 수 있는 '통합 데이터 공간(Integrated Data-space)'인 Dataportal을 구축했습니다.

    • ① 통합 검색 (Unified Search): 구글과 같은 미니멀하고 빠른 검색 경험을 제공합니다.

      • 단순 키워드 매칭이 아니라, PageRank 알고리즘을 활용해 문서화가 잘 되어 있고 자주 사용되는 고품질 자산을 검색 상단에 노출합니다.

    • ② 맥락 및 메타데이터 (Context & Metadata):

      • 데이터의 기원, 생성자, 소비 기록, 마지막 업데이트 시점 등 풍부한 메타데이터를 제공합니다.

      • 리니지(Lineage): 상위/하위 테이블 관계를 시각화하여 데이터 간의 종속성을 파악하게 함으로써 데이터에 대한 신뢰를 높입니다.

    • ③ 사용자 중심 데이터 (Employee-centric):

      • "사람도 데이터 자산"이라는 관점에서, 특정 직원이 어떤 데이터를 만들고 소비했는지 보여주는 사용자 페이지를 제공하여 구전 지식을 시스템화했습니다.

    • ④ 팀 중심 데이터 (Team-centric):

      • 각 팀이 자주 사용하는 테이블과 대시보드를 큐레이션하여, 팀 내외 사용자들이 필요한 정보에 빠르게 접근하도록 돕습니다.

4. 기술 및 UX 전략 (Tech & UX Insights)

  • 그래프 모델(Graph Model): 에어비앤비는 데이터 생태계를 '노드(테이블, 대시보드, 유저)'와 '관계(소비, 생성, 연결)'로 이루어진 그래프로 정의했습니다. 이를 위해 **Neo4j(그래프 DB)**와 **Elasticsearch(검색 엔진)**를 결합하여 복잡한 데이터 간의 연결 고리를 성능 저하 없이 탐색할 수 있게 했습니다.

  • UX 우선주의: 데이터 도구의 인터페이스가 투박해서는 안 된다는 철학 아래, 데이터 과학자와 시각화 엔지니어를 투입하여 직관적이고 버그 없는 프론트엔드(React, Redux 활용)를 구축했습니다. 사용자가 도구를 신뢰해야 데이터도 신뢰하기 때문입니다.

5. 시사점: Atlan과 같은 현대적 솔루션으로의 확장

에어비앤비처럼 막대한 리소스를 투입해 자체 구축(Build)하기 어려운 조직의 경우, Atlan과 같은 'Active Metadata Platform'이 대안이 됩니다. Atlan은 에어비앤비가 지향했던 데이터 민주화 가치를 SaaS 형태로 제공하며, 특히 AI 에이전트와의 결합 및 자동화된 컬럼 레벨 리니지 캡처를 통해 탐색 비용을 획기적으로 낮춰줍니다. (atlan)

마케터 (Phase 1 핵심 사용자)

업무 장면: 리타겟팅 캠페인을 만들 때 Braze에서 속성 이름을 하나씩 클릭해 추측하고, Slack으로 DA에게 문의한 후 최대 하루를 기다려야 세그먼트를 확정할 수 있다. 새 속성 하나를 쓰는 데 평균 4시간이 걸린다.

핵심 니즈:

  • 속성 설명 (한국어): 영문 이벤트명으로는 의미 파악 불가. 한국어 설명 필수
  • 예시값: "category_id가 123이면 아우터인지 슈즈인지 알 수 없다" — 실제 수집되는 값 예시 필요
  • 수집 상태: 지금도 이 속성이 수집되고 있는지 "정상 수집 중 ✅" 처럼 해석된 상태로 노출
  • 활용 현황: 어느 코호트·캠페인에서 이미 쓰이는지 — 이 정보가 Amplitude Catalog보다 MATCH에서 더 중요한 이유

Phase 1 목표: 새 속성 탐색·검증 시간 4시간 → 30분 이내. DA Slack 문의 50% 감소.

운영자 (Phase 1 부분 지원, Phase 2 메인 사용자)

업무 장면: 신규 속성이 실제로 수집되는지 확인하려면 ClickHouse 쿼리를 직접 돌리거나 DA에게 요청해야 한다. 속성이 오디언스·캠페인에 연결된 구조가 없어, 속성 삭제·변경 시 파급 범위를 알 수 없다.

핵심 니즈:

  • 수집 이상 감지: 급감·유실·스키마 불일치를 자동으로 감지해 알림
  • 변경 이력: 언제 어떤 속성이 변경됐는지 이력 조회
  • Deprecation 관리: 속성 폐기 전 영향 범위(연결된 오디언스·캠페인 수) 사전 파악
  • 주의: 수집 현황 숫자(raw volume)를 마케터에게 그대로 노출하면 혼란 — 운영자용 상세 뷰와 마케터용 상태값 뷰를 분리 설계 권장

엔지니어 (Phase 1 기본 지원)

업무 장면: 새 이벤트를 배포할 때마다 명세서를 수동으로 업데이트하고 Confluence에 올린다. 실제 수집된 값과 명세서가 어긋나도 누군가 Slack에 문의하기 전까지 알 수 없다.

핵심 니즈:

  • 스키마 정보: 데이터 타입(string/int/bool), Nullable 여부, 원천 시스템 표기
  • 자동 동기화: 실제 수집 데이터와 카탈로그 명세를 일 1회 자동 동기화 — 수동 업데이트 부담 제거
  • 스키마 변경 감지: 신규·변경·삭제 이벤트 발생 시 자동 알림 (E4-2, Phase 2)

일반 사업부 (Phase 2 이후 대상)

업무 장면: 데이터 활용을 기획하고 싶지만 어떤 데이터가 있는지 파악하는 창구가 없다. DA나 마케팅팀에 직접 물어봐야 한다.

핵심 니즈:

  • 데이터 존재 파악: 전문 기술 없이 "이런 데이터가 있다"를 브라우징 가능한 뷰
  • 담당자 연결: 데이터 관련 문의 시 누구에게 연락해야 하는지 (Data Steward, E2-5)
  • 활용 가능성: 이 데이터를 어떤 목적으로 활용할 수 있는지 사용 사례 노출

Phase 2 이후 고려사항: 일반 사업부의 탐색 패턴은 마케터와 달리 카테고리·태그 중심이므로, E2-2 텍소노미 표준화가 완성된 이후에 의미있는 경험 제공 가능.

데이터 리니지(Data Lineage): 흐름의 가시화와 자동화

  • 데이터 리니지는 데이터가 소스에서 출발해 변환 과정을 거쳐 최종 목적지에 도달하기까지의 흐름을 추적하고, 시스템 전반의 기원, 변경 사항 및 종속성을 문서화하는 기술입니다. 이는 크게 두 가지 방향으로 관리됩니다.

    • 상류(Upstream) 및 하류(Downstream) 관리: 내가 보고 있는 데이터의 기원(Upstream)을 파악하여 신뢰도를 검증하고, 이 데이터가 어디에 쓰이고 있는지(Downstream)를 파악하여 변경 시 영향도를 분석합니다.

    • 리니지는 데이터 구조 변경 시 장애를 예방하는 영향도 분석(Impact Analysis), 지표 이상 시 원인을 찾는 디버깅(Root Cause Analysis), 그리고 중복 데이터 생성을 막는 비용 최적화의 핵심 도구가 됩니다.
